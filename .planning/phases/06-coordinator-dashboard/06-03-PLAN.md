  <name>Task 1: Create status update endpoint</name>
  <files>server/api/issues/[id]/status.patch.ts</files>
  <action>
    Create PATCH /api/issues/[id]/status endpoint for updating issue status:

    1. Check authentication (use existing auth middleware pattern)

    2. Read body with Zod validation:
       - schema: { status: z.enum(['open', 'in-progress', 'resolved', 'closed']) }

    3. Find issue by id, return 404 if not found

    4. Validate status transition (only allow valid workflow):
       - open → in-progress ✓
       - open → resolved ✓
       - open → closed ✓
       - in-progress → resolved ✓
       - in-progress → closed ✓
       - resolved → closed ✓
       - resolved → in-progress ✓ (reopen)
       - closed → in-progress ✓ (reopen)
       - resolved → open (regression)
       - closed → open (regression)

    5. Update issue.status = newStatus, issue.updatedAt = now()

    6. Return updated issue

    Add validation error for invalid transitions with helpful message.
  </action>
  <verify>
    - Valid status transitions work
    - Invalid transitions return 400 with error message
    - Issue updatedAt timestamp updates correctly
    - Non-existent issue returns 404
    - Unauthenticated request returns 401
  </verify>
  <done>Status update endpoint with workflow validation</done>
</task>

<task type="auto">
  <name>Task 2: Create assignment endpoint</name>
  <files>server/api/issues/[id]/assign.patch.ts</files>
  <action>
    Create PATCH /api/issues/[id]/assign endpoint for assigning coordinators to issues:

    1. Check authentication

    2. Read body with Zod validation:
       - schema: { assignedTo: z.array(z.string()) } (array of coordinator IDs)

    3. Find issue by id, return 404 if not found

    4. Validate all coordinator IDs exist in Coordinator collection
       - Query: Coordinator.find({ _id: { $in: assignedTo }, active: true })
       - Return 400 if any coordinator IDs are invalid

    5. Update issue.assignedTo = assignedTo array, issue.updatedAt = now()

    6. Return updated issue with populated coordinator details

    7. Optional: Send notification to newly assigned coordinators (skip for now, Phase 5 will handle notifications)
  </action>
  <verify>
    - Valid coordinator IDs update assignment correctly
    - Invalid coordinator IDs return 400 error
    - Empty array clears assignment
    - Coordinator details are populated in response
    - updatedAt timestamp updates correctly
  </verify>
  <done>Assignment endpoint with coordinator validation</done>
</task>
