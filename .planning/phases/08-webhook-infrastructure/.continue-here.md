<decisions_made>

**Phase 8.1 (Webhook Event Logging):**
- WebhookEvent model with unique eventId constraint for idempotency - prevents duplicate processing
- Event status lifecycle (pending → processing → completed/failed) for monitoring
- 30-day TTL index balances debugging retention with storage management
- Six indexes for optimal query performance

**Phase 8.2 (Stripe Webhook Endpoint):**
- Use readRawBody() NOT readBody() - parsing breaks signature verification
- Database-backed idempotency via WebhookEvent.findOne({ eventId }) lookup
- Duplicate events return 200 OK immediately to prevent Stripe retry loops
- Event router with 13 stub handlers for all required Stripe event types

**Phase 8.3 (Webhook Event Handlers):**
- Handler-per-event pattern for clean separation of concerns
- handleCheckoutSessionCompleted updates Payment status to 'succeeded' and enables Plan.canPrint
- handleSubscriptionCreated enables Plan.qrEnabled for QR issue access
- handleSubscriptionDeleted disables Plan.qrEnabled when subscription canceled
- All handlers validate required metadata (userId, planId) before database operations

**Architecture decision:**
- Separate webhookHandlers.ts utility file for clean handler organization
- Endpoint handles verification + routing, handlers handle business logic
- All database updates in handlers, endpoint only manages event lifecycle

</decisions_made>
