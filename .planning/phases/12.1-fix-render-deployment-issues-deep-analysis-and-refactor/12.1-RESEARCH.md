**Domain:** Nuxt 4 deployment on Render.com
**Confidence:** HIGH

## Summary

The app fails to deploy on Render.com. Based on analysis of the project configuration, the primary issues are likely related to: (1) **Package manager inconsistency** - `package.json` declares `bun` but `render.yaml` uses `npm ci`, (2) **Build memory exhaustion** - Nuxt 4 builds are memory-intensive and may exceed available memory, (3) **Missing build optimization** - The `render.yaml` uses standard `npm run build` instead of the optimized `build:render` script with memory limits, (4) **Potential Sharp native module issues** with externalized configuration.

**Primary recommendation:** Standardize on `npm` for Render deployment (simpler CI/CD), use the optimized `build:render` script, ensure proper memory configuration, and verify all environment variables are set.

### Pattern 1: Render Build Optimization
**What:** Configure memory limits and build caching
**When to use:** All Nuxt 4 deployments on Render
**Example:**
```yaml
# render.yaml
buildCommand: npm ci --prefer-offline --no-audit && NODE_OPTIONS=--max-old-space-size=4096 npm run build
```

**Source:** Render docs (docs.render.com/deploys) + Nuxt docs (nuxt.com/docs/getting-started/deployment)

### Pattern 2: SSR: false with Node-Server Preset
**What:** Client-side rendering with Node.js server for API routes
**When to use:** When SSR is disabled but server API routes are needed
**Example:**
```typescript
// nuxt.config.ts
export default defineNuxtConfig({
  ssr: false,
  nitro: {
    preset: 'node-server',
    port: parseInt(process.env.PORT || '3000')  // Render uses PORT env var
  }
})
```

**Source:** Nuxt deployment docs + Nitro node-server preset docs

### Anti-Patterns to Avoid
- **Mixed package managers:** Don't use bun in package.json but npm in CI/CD
- **Missing NODE_OPTIONS:** Nuxt 4 builds need memory limit configuration
- **Wrong port binding:** Render expects `0.0.0.0` on `PORT` (default 10000)

## Common Pitfalls

### Pitfall 1: Package Manager Mismatch
**What goes wrong:** `package.json` declares `bun` but `render.yaml` uses `npm ci`, causing lock file issues
**Why it happens:** Inconsistent configuration between development and deployment
**How to avoid:** Standardize on npm for both environments, remove bun-specific config
**Warning signs:** "lockfile version mismatch" errors, dependency resolution failures

### Pitfall 2: Build Memory Exhaustion
**What goes wrong:** JavaScript heap out of memory during `nuxt build`
**Why it happens:** Nuxt 4 + Vite builds are memory-intensive, default Node memory limit (1.4GB on 64-bit) is insufficient
**How to avoid:** Set `NODE_OPTIONS=--max-old-space-size=4096` before build command
**Warning signs:** "CALL_AND_RETRY_LAST Allocation failed", build timeout at 90%+

### Pitfall 3: Port Binding Issues
**What goes wrong:** App fails health check, shows "Failed to bind to port"
**Why it happens:** Not respecting Render's `PORT` environment variable
**How to avoid:** Configure Nitro to read `PORT` env var, default to 10000
**Warning signs:** 502 Bad Gateway, health check timeout

### Pitfall 4: Sharp Native Module Issues
**What goes wrong:** "Cannot find module sharp" or native binding errors
**Why it happens:** Sharp has platform-specific native binaries
**How to avoid:** Keep `sharp` in `externals.external` array (already done), ensure npm rebuilds
**Warning signs:** Module resolution errors at startup

### Pitfall 5: Environment Variable Missing
**What goes wrong:** App crashes on startup with "JWT_SECRET is required" or similar
**Why it happens:** Required env vars not set in Render dashboard
**How to avoid:** Audit all env vars in `nuxt.config.ts` runtimeConfig, set in Render
**Warning signs:** Immediate crash on startup, undefined errors
